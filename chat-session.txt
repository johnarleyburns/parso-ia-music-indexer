Standard Collaborative Filtering (CF)—recommending tracks based purely on what similar users listen to—will fail here because of the "cold start" problem. The vast majority of IA's millions of tracks have zero listening history in your app.

To solve this, we need to architect a Hybrid Music Recommendation System that analyzes the audio files themselves before anyone even listens to them. Here is a blueprint for how we can structure the data pipeline, the backend, and the iOS client experience.

1. The Audio Ingestion Pipeline (Content-Based Filtering)
Instead of relying on user tags, the engine must "listen" to the files. We can build an autonomous backend worker in Golang to concurrently stream, sample, and analyze chunks of IA audio files.

For each track, the pipeline should extract Mel-frequency cepstral coefficients (MFCCs). MFCCs mathematically represent the short-term power spectrum of a sound. By using a deep learning model to process these coefficients, the system can automatically extract high-level concepts like tempo, genre, and instrumentation directly from the audio wave (Wang & Wang, 2014).

2. The Logistics: IA Rate Limits & Throttling
The Internet Archive is a non-profit library, not a CDN, and they actively throttle bulk downloading to protect their servers.  
Nieman Lab

API Rate Limits (Metadata): For querying the Advanced Search API to get file lists and metadata, community consensus is that IA soft-bans or heavily rate-limits IPs exceeding 15 requests per minute. You must implement a strict sleep() or use a token bucket rate limiter in your Golang workers (e.g., 1 request every 4-5 seconds) just to get the URLs.

Download Throttling (Audio): IA throttles actual file downloads per IP. Users frequently report being capped at 500 KB/s to 1 MB/s, regardless of gigabit fiber connections. Furthermore, opening too many concurrent connections from a single IP will cause all of them to slow to a crawl, or result in HTTP 429 (Too Many Requests) and HTTP 503 errors.

3. The Strategy: HTTP Range Requests (Partial Streaming)
Downloading 5 million full FLAC or MP3 files at 500 KB/s would take literal decades and petabytes of storage.

The Solution: Do not download the files. Stream them.
Your Golang ingestion workers should use HTTP Range requests (Range: bytes=0-X) to pull only the first 30 to 45 seconds of the VBR MP3 derivative file (IA automatically generates smaller MP3s for almost all FLAC uploads).

Stream to Memory: The Go worker requests the first ~600KB of the MP3 file directly into memory.

Analyze in RAM: The ML library extracts the MFCCs and noise floor data directly from the memory buffer.

Discard: The buffer is immediately flushed. No audio is ever written to disk.

This drastically reduces the bandwidth footprint and completely eliminates the need for massive local storage arrays.

If a worker crashes, the track remains "processing". A simple cleanup script can periodically reset any tracks stuck in "processing" for more than 5 minutes back to "pending".

The challenge with Golang is that it does not have a monolithic ML library like Python’s librosa. However, we can build a highly efficient, modular pipeline by composing a few specific Go packages:

Feature Extraction: github.com/zrma/go-mfcc (A highly optimized Go library specifically for computing MFCCs).

Vector DB: github.com/mattn/go-sqlite3 combined with the sqlite-vec C-extension.

2. The MFCC Extraction Logic
When the Go worker streams the 30-second chunk, it pulls down a compressed MP3.

go-mp3 decodes that chunk into raw PCM audio data in RAM.

We pass the PCM data to go-mfcc, configuring it to extract 20 Mel bands. This generates a 2D array (20 bands × thousands of time frames).

A database cannot efficiently search a 2D matrix. The Go worker must calculate the mean and variance for each of the 20 bands across the time axis. This flattens the matrix into a dense, 40-dimensional []float32 vector, which perfectly captures the track's sonic "fingerprint" (tempo, timbre, and instrumentation) in a format sqlite-vec can instantly query.

However, executing this across 4+ million files introduces a massive architectural hurdle. Here is how we avoid the pagination trap and orchestrate the workload across multiple servers.

1. The 10,000 Limit Trap (and the Cursor Solution)
If you try to use standard offset pagination (e.g., &rows=1000&page=15) with the Internet Archive's Advanced Search API, your code will violently crash once you hit the 10,000th result. The Archive's Lucene backend strictly rejects deep pagination on sorted queries to protect their memory.

The Solution: You must use the Internet Archive Scraping API.
Instead of passing page or offset, the Scraping API gives you a cursor. You pass the cursor back in your next request to get the next block of items. It fully supports sorts=downloads desc and allows you to cleanly walk through all 4 million items without hitting a depth cliff.

2. Decoupling Search from Extraction (The Master/Worker Pattern)
You should not have your distributed worker servers querying the IA search API directly. If multiple servers are trying to search and paginate simultaneously, they will clash, overlap, and trip IA's rate limits.

We need to decouple the "Discovery" from the "Analysis".

Step A: The Coordinator (Single Process)
You run one single, lightweight Go script (or even just a Python script) on your main server. Its only job is to walk the Scraping API using the cursor.

It hits the Scraping API: query=(collection:etree OR collection:netlabels) AND mediatype:audio&sorts=downloads desc.

It takes the returned JSON of IDs and pushes them into your database's catalog_queue table.

It takes the cursor string and makes the next request.
Because it is only downloading text metadata, one server can index the metadata for all 4 million items into your database queue in a matter of hours.

Step B: The Fleet (Distributed Workers)
Now, your catalog_queue is populated, ordered naturally by download count descending.
This is where your multiple servers come in. Your distributed Golang workers do not talk to the IA Search API. They only talk to your database.

Worker A (IP address 1) connects to the database, claims 50 pending rows, and locks them.

Worker B (IP address 2) connects, claims the next 50 pending rows, and locks them.

The workers stream the MP3s via the specific item URLs, extract the MFCCs, and write the embeddings back to the database.

3. State Management Across Servers
If you are running multiple workers across different networks, a local SQLite file won't work anymore. You will need a central database to track the state.

Since you prefer the autonomy and low overhead of SQLite, the modern agentic approach is to use Turso. Turso is a distributed database built on libSQL (a fork of SQLite).

It feels and codes exactly like SQLite.

It supports vector embeddings natively.

You host the primary database, and your Go workers connect to it via HTTP, allowing them to coordinate the queue safely from anywhere in the world.

1. The Per-Row Math
Each entry in your track_embeddings table consists of:

MFCC Embedding: 40 dimensions of FLOAT32 (4 bytes each).

40×4 bytes=160 bytes.

Quality Score: One REAL (8 bytes).

SQLite/Vector Overhead: Database page headers, row pointers, and internal B-Tree fragmentation typically add ~20-30% to the raw data size.

Estimated Raw Row Size: 30+160+8=198 bytes.
Including 25% Overhead: ≈250 bytes per row.

Note: These estimates assume you are storing raw FLOAT32 vectors. If you eventually use vector quantization (e.g., INT8 or BIT packing) as your database grows to millions of rows, you could reduce the storage footprint of the vectors themselves by 4x to 32x.

3. Key Architectural Implications
Memory vs. Disk: Because you are using sqlite-vec (via Turso/libSQL), the entire 1 GB database file will easily reside on disk. During search queries, Turso will pull the necessary pages into memory. A 1 GB index is very small by modern server standards; you will not need massive RAM, and search queries will be nearly instantaneous.

The "One-Gigabyte" Advantage: Storing 4 million high-quality audio fingerprints in only 1 GB of space is incredibly efficient. This means the entire database can be cached in the memory of even the smallest, cheapest cloud instances (e.g., a $5/month VPS), leading to sub-millisecond search speeds.

Scaling Distributed Workers: Since the total size for 4 million tracks is only ~1 GB, you do not need to worry about storage bottlenecks when using multiple servers. You can easily replicate this database across multiple edge locations or nodes without running into massive synchronization or storage costs.
